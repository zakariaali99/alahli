# Decisions Made

- **User photo on User model directly** — simplest approach, avoids separate Trainer model, works with existing group coach assignment
- **Staff management as single table** — one combined page with role filter instead of per-tab views
- **Admin add-athlete as direct creation** — no pending step (admin is trusted), bypasses registration approval
- **Form inputs use only standard Tailwind utility classes** — no custom CSS classes or raw `hsl(var(...))` because the project's CSS variables are plain hex colors, not HSL values
- **Role-based auth only** — no per-page ACL, just role checks at the API level
- **Website registration no longer auto-creates Athlete** — admin must manually create from dashboard via the new create-athlete action

# Problems Encountered

1. **Blank pages on login** — auth redirect returned `null` instead of `<Navigate>` component, causing React to render nothing
2. **Duplicate route definition** — `notifications` path appeared twice in `App.tsx`, causing one to silently fail
3. **Logout listener leak** — `AuthProvider` never cleaned up its event listener
4. **Invisible form inputs** — 4 admin pages used embedded `<style>` blocks with `hsl(var(--border, 220 13% 91%))` but the project's CSS variables are plain hex (e.g., `--border: #c8ccd8`), making `hsl(#c8ccd8)` invalid CSS
5. **Stale closure in SubscriptionPage** — `setData(prev => ...)` not used in some places, causing state corruption on rapid step transitions
6. **Missing athleteId for self-subscribing athletes** — the subscription endpoint didn't receive athlete ID when the authenticated user was themselves an athlete
7. **Subscription API key mismatch** — `NewAthletes.tsx` looked up subscriptions via `sub.athlete_name` but the API returns `sub.athlete`
8. **Paginated API only returned first page** — `AdminNotifications.tsx` fetched active/expired subscriptions but never looped through subsequent pages
9. **QR scanner unreliable** — the zxing-based scanner had inconsistent decode; replaced with html5-qrcode which has a proper continuous decode loop
10. **Hardcoded hex colors in Login.tsx** — used `#d8e1ea`, `#f9fbfd`, `#2a3f55`, `#6a7f95`, `#0F4C81` instead of design tokens, breaking theme consistency

# Important Context

- **Root cause of invisible borders**: Project CSS variables (`--border`, `--primary`, `--surface-container-low`) are defined as plain hex colors (e.g., `--border: #c8ccd8`). The admin pages used `hsl(var(--border, 220 13% 91%))` in embedded `<style>` blocks, but `hsl(#c8ccd8)` is invalid CSS and produces no visible border, background, or focus ring.
- **Standard Tailwind input class that works project-wide**: `w-full bg-surface-container-low border border-border rounded-xl px-3 py-2.5 text-sm text-foreground outline-none focus:ring-2 focus:ring-primary focus:border-primary transition-colors` (with `appearance-none cursor-pointer` for selects, and `file:` utilities for file inputs).
- **Website registration no longer auto-creates `Athlete`** — admin must manually create from dashboard via the new `create-athlete` action.
- **Approve button blocked** until athlete profile exists for athlete-type registrations.
- **Old registration records** (created before this change) already have linked athletes — backward compatible.
- **Build warning**: chunk `vendor-charts` > 500 kB after minification — non-blocking Vite warning.
- **Auth tokens**: JWT stored in `localStorage`; `Bearer` prefix used in API requests; refresh endpoint at `/api/token/refresh/`.
- **Arabic locale**: `ar-SA-u-nu-latn` for Latin digits in Arabic context.
